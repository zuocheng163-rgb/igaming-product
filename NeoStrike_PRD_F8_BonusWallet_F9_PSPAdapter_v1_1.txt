F8: Bonus Wallet Module



This PRD specifies two features that together close NeoStrike's final competitive gap against one-stop-shop platforms such as Softswiss and EveryMatrix for greenfield operators.

F8 — Bonus Wallet Module: implements the three Fast Track bonus endpoints (GET /bonus/list, POST /bonus/credit, POST /bonus/credit/funds) and all underlying bonus execution mechanics. This is a mandatory Fast Track integration requirement — without it NeoStrike cannot claim to be a complete Fast Track integration.

F9 — PSP Adapter BYOC: implements a standardised payment adapter interface. Operators bring their own licensed PSP contract. NeoStrike provides the integration layer, captures structured payment events, and publishes them to Fast Track via RabbitMQ. NeoStrike never touches settlement funds.

BUSINESS POSITIONING: Both features are scoped as Technical Service Provider capabilities. NeoStrike never issues bonuses on its own initiative, never holds a merchant account, and never assumes chargeback liability. All decisions remain with the operator and their licensed partners. NeoStrike executes and logs.


1. Strategic Context

1.1 The Competitive Gap
NeoStrike's 2-week launch promise and serverless cost model are compelling for greenfield Type A and Type B operators. However, these operators also need a bonus engine and payment processing to run a viable casino. Without these, they must choose a bundled platform (Softswiss, EveryMatrix, Pragmatic Solutions) that includes these capabilities — despite much higher GGR-based fees.

F8 and F9 close this gap entirely. With both features live, a greenfield operator gets game providers, payments, bonus campaigns, CRM, and responsible gambling compliance from a single €54–79/month serverless stack.

1.2 Fast Track as the Bonus Brain
Fast Track Rewards (powered by the January 2026 Greco acquisition) handles campaign logic: which players qualify for a bonus, when, and why. Fast Track communicates bonus decisions to NeoStrike via three defined API endpoints. NeoStrike executes the bonus mechanics — balance crediting, wagering tracking, and fund release — and publishes outcomes back to Fast Track for Data Studio reporting.

1.3 Business Risk Position
NeoStrike's TSP (Technical Service Provider) position means it never faces the player and holds no gambling licence. The operator is accountable to regulators for all bonus and payment outcomes. NeoStrike's liability is purely contractual — limited to software correctness, which is managed through immutable audit logging and idempotent API design.

Fast Track's own API design reinforces this: the X-Fasttrack-Id header acts as an idempotency key, meaning every bonus credit request is traceable to a specific Fast Track campaign action. NeoStrike executed this request exactly once at this timestamp and returned this result. That is NeoStrike's complete defence against any operator dispute.


  Feature F8: Bonus Wallet Module

Implements the Fast Track bonus API contract and all underlying bonus execution mechanics. Makes NeoStrike a complete Fast Track integration.

2.1 Problem Statement
NeoStrike's existing wallet tracks real balance only. When Fast Track calls /bonus/credit, NeoStrike currently has no endpoint to receive it — the request fails with 404. This means operators cannot run any Fast Track campaign that awards a bonus, which covers welcome bonuses, deposit matches, free spin credits, loyalty rewards, and cashback. Without F8, NeoStrike operators cannot use Fast Track Rewards at all.

2.2 Goals & Non-Goals
Goals
	•	Implement all three Fast Track bonus endpoints: GET /bonus/list, POST /bonus/credit, POST /bonus/credit/funds
	•	Maintain a separate bonus balance ledger alongside real balance in Supabase
	•	Track wagering requirement progress per player per active bonus in real time
	•	Enforce bonus rules: wagering requirements, game contribution rates, win cap, expiry
	•	Release bonus funds to real balance automatically when wagering is complete
	•	Store all Fast Track correlation headers for campaign cost reporting in Fast Track Data Studio
	•	Log every bonus event immutably for operator dispute resolution and regulatory audit

Non-Goals
	•	NeoStrike will not decide which players receive bonuses — that is Fast Track's domain
	•	NeoStrike will not build a campaign management UI — operators use Fast Track back-office
	•	NeoStrike will not implement free spins mechanics — free spin round values are credited as bonus balance
	•	NeoStrike will not support bonus chaining (bonus triggering another bonus) in v1

2.4 Bonus Execution Mechanics
Bonus Balance Ledger
NeoStrike maintains two wallet balances per player: real_balance (existing) and bonus_balance (new). They are stored as separate columns in the existing player_wallets table. The combined_balance exposed in API responses equals real_balance + bonus_balance but the two are tracked separately.

Wagering order: real balance is always staked first. Bonus balance is only staked when real balance reaches zero. This is the standard iGaming wagering order and is non-configurable in v1.

Bonus Instance Lifecycle
State
Trigger
Action
PENDING
POST /bonus/credit received, awaiting processing
Bonus request received from Fast Track. Idempotency check in progress. Transitions to CREATED on successful balance credit.
CREATED
Bonus balance successfully credited
Bonus instance created. Bonus balance credited to player wallet. Wagering rules loaded. BONUS_AWARDED event published to RabbitMQ. Awaiting first qualifying wager to transition to ONGOING.
ONGOING
First qualifying wager placed
Wagering progress updated on every debit callback that matches game contribution rules. Win cap monitored on every credit callback. Transitions to COMPLETED when wagering_progress >= wagering_required, or to FORFEITED if win cap is hit.
COMPLETED
Wagering requirement met (wagering_progress >= wagering_required)
Bonus balance released in full to real balance. BONUS_CONVERTED event published to RabbitMQ. Terminal state.
FORFEITED
Operator cancels via Tenant Portal, OR player winnings exceed win_cap
Bonus balance forfeited. On win cap: winnings up to win_cap released to real balance, remainder forfeited. On operator cancel: full bonus balance forfeited. BONUS_FORFEITED event published. Mandatory reason field logged. Terminal state.
EXPIRED
expiry_days elapsed without reaching COMPLETED
Remaining bonus balance forfeited. BONUS_EXPIRED event published to RabbitMQ. Real balance unaffected. Terminal state.

Wagering Requirement Tracking
On every debit callback, NeoStrike calculates the wagering contribution using the game's contribution rate defined in the bonus template. The contribution is added to wagering_progress. When wagering_progress >= wagering_required, the bonus transitions to COMPLETED.

wagering_contribution = bet_amount × game_contribution_rate
wagering_progress    += wagering_contribution
if wagering_progress >= wagering_required: → release bonus to real balance

Win Cap Enforcement
When a win credit arrives and the player has an active bonus, NeoStrike checks whether total winnings on the active bonus would exceed win_cap. Winnings above win_cap are credited to real balance. Winnings up to win_cap accumulate in bonus balance and are subject to wagering requirements. If win_cap is reached, the bonus transitions to FORFEITED state.

2.8 Compliance Considerations
UKGC — 10x Wagering Cap: UKGC regulations require that bonus wagering requirements do not exceed 10x the bonus amount. The Tenant Portal must enforce a maximum wagering_requirement of 10 for UK-licensed tenants. Attempts to set higher values are blocked with a clear error message referencing the UKGC requirement.

Germany / Netherlands — Bonus Restrictions: Some regulated markets restrict bonus types by payment method or prohibit bonus advertising entirely. The PSP Adapter (F9) provides payment method data that allows the Bonus Wallet Module to enforce payment-method-conditional bonus eligibility rules.

  Feature F9: PSP Adapter (BYOC)

Standardised payment adapter interface. Operators bring their own licensed PSP contract. NeoStrike captures payment events, updates the wallet, and publishes to Fast Track. NeoStrike never touches settlement.

3.1 Problem Statement
NeoStrike's current wallet has no payment processing layer. Operators must handle deposits and withdrawals entirely outside NeoStrike, which means: deposit events are not captured with payment method metadata, Fast Track does not receive structured DEPOSIT events with PSP data, and the Bonus Wallet Module cannot enforce payment-method-conditional bonus rules. F9 closes all three gaps.

SCOPE BOUNDARY: NeoStrike is not a payment facilitator, payment processor, or Merchant of Record. NeoStrike never holds settlement funds, never registers with card schemes, and never assumes chargeback liability. The operator and their PSP own the payment relationship entirely. NeoStrike's role is event capture, wallet update, and RabbitMQ relay.

3.2 BYOC Architecture
The operator contracts their own PSP directly. The PSP is configured to send deposit and withdrawal confirmation webhooks to NeoStrike's PSP Adapter endpoint. NeoStrike validates the webhook, credits or debits the player wallet, and publishes a structured event to RabbitMQ. The PSP handles all settlement, refunds, and chargeback management independently.

3.3 PSPAdapter Interface
interface PSPAdapter {
  initiateDeposit(req: DepositRequest):      Promise<DepositInitResponse>;
  initiateWithdrawal(req: WithdrawRequest):  Promise<WithdrawInitResponse>;
  verifyWebhook(payload: string,             
                signature: string): boolean; // Webhook HMAC verification
  normaliseEvent(raw: unknown):              NormalisedPaymentEvent;
  healthCheck():                             Promise<HealthStatus>;
}

All PSP adapters implement this interface. NeoStrike's payment processing logic is adapter-agnostic — swapping PSP requires only a new adapter class. The operator selects their PSP connector in Tenant Portal and provides credentials. No code changes required.

3.4 Normalised Payment Event Schema
Every PSP adapter normalises its raw webhook payload into NeoStrike's standard payment event schema before wallet update and RabbitMQ publish. This schema is designed to include all fields Fast Track needs for campaign segmentation.

{
  "event_type":     "DEPOSIT",            // DEPOSIT | WITHDRAWAL | REFUND | CHARGEBACK
  "transaction_id": "neostrike_pmt_abc",  // NeoStrike-generated idempotency key
  "psp_tx_id":      "nuvei_tx_123",       // PSP-side transaction reference
  "player_id":      "player_123",
  "amount":         100.00,               // Gross amount
  "net_amount":     98.50,                // After PSP fees
  "currency":       "EUR",
  "payment_method": "visa_credit",        // Normalised method identifier
  "vendor_id":      "nuvei",              // PSP identifier for Fast Track
  "vendor_name":    "Nuvei",
  "payment_id":     "nuvei_tx_123",       // For Fast Track Data Studio
  "status":         "CONFIRMED",          // CONFIRMED | PENDING | FAILED
  "timestamp":      "2026-02-17T15:30:00Z"
}

The payment_method field uses a normalised identifier taxonomy across all PSP adapters. This ensures Fast Track campaign rules based on payment method work correctly regardless of which PSP the operator uses.

3.6 Deposit Initiation API
NeoStrike exposes a deposit initiation endpoint that the operator's frontend calls to start a deposit flow. NeoStrike calls the PSP, gets a checkout URL or session token, and returns it to the frontend. The frontend redirects the player to the PSP checkout. All subsequent steps are PSP-to-NeoStrike server-to-server.

POST /api/v1/payments/deposit/initiate
Authorization: Bearer {NEOSTRIKE_API_KEY}
{
  "player_id": "player_123",
  "amount":    100.00,
  "currency":  "EUR",
  "return_url":"https://mycasino.com/deposit/complete"
}

// Response 200 OK
{
  "transaction_id":  "neostrike_pmt_abc",
  "checkout_url":    "https://checkout.nuvei.com/...",
  "expires_at":      "2026-02-17T16:00:00Z"
}

The operator's frontend redirects to checkout_url. The player completes payment on the PSP's hosted page. NeoStrike receives the confirmation webhook from the PSP and credits the wallet. The operator's frontend polls GET /api/v1/payments/status/{transaction_id} to update the UI.

3.7 Withdrawal Flow
Withdrawal initiation follows the same pattern. The operator's frontend calls POST /api/v1/payments/withdrawal/initiate. NeoStrike debits the player's real balance immediately (pending state), calls the PSP to initiate the payout, and publishes a WITHDRAWAL event to RabbitMQ. PSP sends a confirmation or failure webhook. On confirmation, withdrawal moves to CONFIRMED state. On failure, the deducted balance is restored and a WITHDRAWAL_FAILED event is published.

3.8 Webhook Security
	•	All PSP webhooks verified using PSP-specific signature method before processing
	•	Nuvei: HMAC-SHA256 of sorted request parameters using Merchant Secret Key
	•	Generic adapter: configurable HMAC-SHA256 or RSA signature verification
	•	Replayed webhooks detected via PSP transaction ID idempotency check in Supabase
	•	Webhook source IP validated against PSP's published IP range (where PSP provides this)
	•	All raw webhook payloads logged to psp_webhook_log table for 90 days

6. Acceptance Criteria

6.1 F8 — Bonus Wallet Module
Fast Track API Contract
	•	GET /bonus/list returns all active bonus templates for the tenant within 10ms (cached)
	•	POST /bonus/credit with a new X-Fasttrack-Id credits bonus_balance and returns 200 within 100ms
	•	POST /bonus/credit with a duplicate X-Fasttrack-Id returns the original 200 response — never credits twice
	•	POST /bonus/credit/funds credits balance immediately and returns 200 within 80ms
	•	All five Fast Track correlation headers stored on every bonus_instance record

Bonus Execution Mechanics
	•	Wagering progress correctly calculated using game contribution rates on every debit callback
	•	Bonus transitions to COMPLETED and releases funds to real_balance when wagering_progress >= wagering_required
	•	Bonus transitions to EXPIRED when expiry_days elapsed without completion — real_balance unaffected
	•	Win cap enforced correctly — winnings above win_cap credited to real_balance, bonus forfeited
	•	Real balance staked before bonus balance on all wagers

Compliance
	•	UK-licensed tenants: Tenant Portal blocks wagering_requirement > 10 with UKGC error message
	•	All bonus events written to bonus_events table — immutable, never updated or deleted
	•	BONUS_AWARDED RabbitMQ event published within 50ms of POST /bonus/credit success

6.2 F9 — PSP Adapter
Nuvei Adapter
	•	POST /payments/deposit/initiate returns Nuvei checkout_url within 500ms
	•	Nuvei deposit confirmation webhook processed within 100ms — real_balance credited correctly
	•	Duplicate Nuvei webhook (same psp_tx_id) detected and ignored — wallet not double-credited
	•	HMAC-SHA256 webhook signature verified — invalid signature returns 401, wallet not updated
	•	DEPOSIT RabbitMQ event includes payment_method, vendor_id, vendor_name, net_amount, payment_id

Generic Adapter
	•	Configurable webhook receiver processes deposit confirmation and credits wallet
	•	Field mapping configuration in Tenant Portal correctly normalises PSP-specific field names

All Adapters
	•	DEPOSIT event triggers Fast Track campaign evaluation — deposit match bonus issued within 2 seconds end-to-end
	•	Withdrawal initiation debits real_balance immediately — balance restored if PSP returns failure
	•	Raw webhook payload stored in psp_webhook_log for 90 days
	•	WITHDRAWAL event published to RabbitMQ within 50ms of webhook confirmation
