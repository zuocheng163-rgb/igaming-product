This document defines the product requirements for two remaining production features of the NeoStrike serverless iGaming middleware platform. Both features are on the critical path to operator onboarding and must meet the 2-week launch promise delivered to customers.

All architecture decisions, technology choices, and behavioural specifications in this document reflect the strategic recommendations agreed during the February 2026 planning session.


1. Executive Summary

NeoStrike is a 100% serverless iGaming middleware platform that enables operators to launch Fast Track-native casinos in 2 weeks instead of the industry-standard 6–8 weeks. Five of seven core features are production-ready. This PRD covers the two remaining features required to fulfil the complete product promise.

2. Feature F6: Game Catalog API

Aggregates game metadata from all enabled providers into a single, cached, searchable endpoint.

2.1 Problem Statement
Without a unified catalog API, every operator must independently integrate each provider's metadata API, handle schema differences, manage caching, and keep metadata current. This represents 2–3 weeks of repetitive boilerplate for every new operator — directly undermining NeoStrike's 2-week launch promise.

2.2 Goals & Non-Goals
Goals
	•	Provide a single endpoint returning normalised game metadata across all enabled providers
	•	Cache responses via Upstash Redis for sub-10ms reads
	•	Support per-tenant game restriction (enable/disable games per operator)
	•	Keep provider metadata current via automated nightly NeoStrike-managed sync
	•	Degrade gracefully when individual providers are offline

Non-Goals
	•	Game session management (handled by POST /launch endpoint)
	•	Player-level personalisation or recommendations (future ML enhancement)
	•	Real-time odds or live data feeds

2.5 Data Architecture
Database Schema
Two tables in Supabase PostgreSQL with Row Level Security enforced at the database level.

  id (PK), provider, name, category, thumbnail,

  tenant_id (FK), game_id (FK), enabled (bool, default TRUE)

2.6 Nightly Metadata Sync
NeoStrike owns and operates the sync process. Operators are never responsible for keeping game metadata current.

	•	Vercel Cron Job fires at 02:00 UTC nightly
	•	For each active tenant, iterate enabled providers (Evolution, Pragmatic, NetEnt)
	•	Pull latest game list from provider metadata API using tenant BYOC credentials
	•	Upsert into games_master table (insert new, update changed, mark removed as inactive)
	•	Invalidate Upstash Redis cache keys for affected tenants
	•	Write sync_log record (tenant_id, provider, games_added, games_updated, duration_ms, status)
	•	On failure: retry 3× with exponential backoff, then alert via Tenant Portal notification

Operators can trigger a Force Sync from Tenant Portal > Game Providers > Sync Now at any time (rate-limited to once per hour per tenant).

2.7 Per-Tenant Game Restrictions
Operators can restrict games by jurisdiction, commercial preference, or lobby curation. The default state is all games enabled — operators only configure exceptions.

	•	Tenant Portal > Game Catalog > Enable/Disable toggle per game
	•	Bulk actions: disable all games from a specific provider or category
	•	Restriction takes effect within 5 minutes (next Redis cache expiry)
	•	Compliance audit log records every restriction change (who, what, when)

All catalog API responses are filtered by tenant_game_config before returning. A disabled game also blocks /launch — returning a 404 game_not_found rather than exposing that the game exists but is restricted.

2.9 Acceptance Criteria
	•	GET /catalog returns correct games for tenant with mixed provider configuration
	•	Disabled game returns 404 on /launch — not 403 (must not leak existence)
	•	Cache miss on cold start resolves in < 200ms
	•	Nightly sync adds a newly published game within 24 hours of provider release
	•	Force Sync in Tenant Portal refreshes catalog within 5 minutes
	•	Provider offline: catalog returns remaining providers' games with provider_status field indicating degraded state
	•	All sync events logged with tenant_id, provider, timestamp, and outcome
	•	RLS in Supabase: tenant A cannot read tenant B game config


3. Feature F7: Reference Frontend + SDK

Open-source React starter kit and npm SDK that eliminates 6–8 weeks of frontend boilerplate for every new operator.

3.1 Problem Statement
Building a casino frontend from scratch requires authentication flows, game lobby architecture, wallet UI, payment integration, responsible gambling popups, and session management. Even experienced teams need 6–8 weeks. NeoStrike's 2-week launch promise is only achievable if operators can fork a production-ready template and brand it — not build from scratch.

3.2 Goals & Non-Goals
Goals
	•	Provide a scaffold CLI (npx @neostrike/create-casino) that bootstraps a complete casino frontend
	•	Publish @neostrike/sdk to npm with React hooks covering all core player actions
	•	Limit operator configuration to 5 environment variables — no provider credentials in the frontend
	•	Support branding customisation via theme.config.js (logo, colours, fonts)
	•	Action-triggered balance refresh only — no polling (cost and architecture consistency)

Non-Goals
	•	Server-side rendering / Next.js (Vite + React SPA, deployable to Vercel Static)
	•	Native mobile apps (responsive web only in v1)
	•	Multi-language i18n (operator responsibility in v1)
	•	Custom game lobby layouts beyond the provided GameLobby component

useBalance — Detail
Balance is NOT polled on an interval. It refreshes in two ways: automatically on every API response (debit, credit, deposit), and manually via the exposed refresh() function. This is consistent with NeoStrike's stateless real-time architecture and eliminates unnecessary Vercel Edge Function invocations.

const { balance, bonusBalance, loading, error, refresh } = useBalance(playerId);
// refresh() — call explicitly after async deposit confirmation
// balance updates automatically after usePayments.deposit() resolves

useGames — Detail
Wraps GET /api/v1/games/catalog. Supports client-side search and filter state, pagination via loadMore(), and deduplication across pages.

const { games, total, loading, search, filter, loadMore } = useGames({
  provider: 'evolution,pragmatic',
  category: 'live-casino',
  limit: 50
});

useAlerts — Detail
Polls the alerts array from the most recent API response. Exposes a dismiss(alertId) function that ACKs the alert server-side (required for UKGC audit compliance — interventions must be acknowledged, not silently ignored).

const { alerts, dismiss } = useAlerts(playerId);
// alerts: Alert[] — includes type, message, severity, action, buttons
// dismiss(alertId) — required for compliance; logs acknowledgement

Environment Variables (.env.example)
Operators configure exactly five variables. No provider credentials, no Supabase keys, no RabbitMQ details — all backend secrets remain in the NeoStrike API layer.

VITE_NEOSTRIKE_API_URL=https://your-api.vercel.app
VITE_NEOSTRIKE_API_KEY=your_tenant_api_key
VITE_CASINO_NAME=My Casino
VITE_DEFAULT_CURRENCY=EUR
VITE_SUPPORT_EMAIL=support@mycasino.com

Branding — theme.config.js
All visual customisation flows through a single config file. Components read from this config at build time — no component code changes required for standard branding.

export default {

3.7 Deployment Flow
	•	Operator runs: npx @neostrike/create-casino my-casino
	•	Fills in 5 environment variables in .env
	•	Edits theme.config.js (logo, colours, casino name)
	•	Runs: vercel deploy — casino live at https://my-casino.vercel.app
	•	Type B: NeoStrike Support handles steps 1–4 on operator's behalf

Total elapsed time from scaffold to live URL: under 10 minutes for a technical operator. Type B operators require no technical involvement beyond reviewing the staging URL and requesting minor tweaks.

3.8 Security Requirements
	•	JWT tokens stored in memory only — never localStorage or sessionStorage (XSS mitigation)
	•	All API calls authenticated via VITE_NEOSTRIKE_API_KEY (tenant-scoped, read-only from frontend)
	•	No provider credentials, database URLs, or RabbitMQ connection strings in frontend build
	•	Content Security Policy headers set by Vercel deployment config
	•	CORS restricted to operator's registered domain (configured in Tenant Portal)
