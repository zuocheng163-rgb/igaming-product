F10 solves the final missing piece in NeoStrike's one-stop-shop stack for greenfield operators: player identity management and KYC compliance. It does so with a clearly bounded GDPR data processor position — NeoStrike stores the minimum player profile data required to serve Fast Track and the operator portal, under the operator's instructions as data controller.

The solution follows the same BYOC (Bring Your Own Contract) philosophy established in F9: operators own and host their own identity infrastructure; NeoStrike provides pre-built connectors that integrate with it. Two connectors ship in v1: Supabase Auth (player authentication) and Sumsub (KYC/AML verification).

GDPR POSITION: NeoStrike is a GDPR data processor for player profile data. The operator is the data controller — they decide what data is collected and why. NeoStrike stores the minimum player profile necessary to serve two legitimate operational requirements: (1) publishing structured player events to Fast Track for campaign execution, and (2) enabling operator support agents to look up players in the Tenant Portal. NeoStrike never stores payment card data, KYC documents, or biometric data. A Data Processing Agreement (DPA) must be signed with each operator before activation. Document scans and personal identity documents remain exclusively within Sumsub's systems, which the operator contracts directly.


1. Strategic Context & GDPR Architecture

1.1 The Final Gap
With F8 (Bonus Wallet Module) and F9 (PSP Adapter) complete, NeoStrike covers game integration, payments, bonus campaigns, and Fast Track CRM. The remaining gap is player identity: authentication, registration, session management, and KYC/AML compliance. Without these, operators cannot onboard players or satisfy regulatory requirements.

Traditional PAMs bundle all of this together. Hosting a full PAM inside NeoStrike would deepen NeoStrike's processor obligations and create scope creep — NeoStrike would become responsible for large volumes of player PII that sit outside its core middleware function. The right approach is to store only what NeoStrike genuinely needs to operate, under a clear DPA with the operator.

1.2 Why a Self-Hosted SDK Is Also Not the Right Answer
A self-hosted PAM SDK would technically remove NeoStrike from the GDPR chain — but it creates a different problem. Greenfield Type A and Type B operators are typically small teams without DevOps capability. Requiring them to deploy, secure, and maintain a custom identity server would create an adoption barrier that undermines NeoStrike's 2-week launch promise.

1.3 The Right Answer: BYOC Identity Connectors
Supabase Auth and Sumsub are best-in-class, well-documented services that operators can be up and running with in hours. The operator contracts them directly and is the GDPR data controller for all player personal information. NeoStrike acts as a data processor — storing the minimum player profile data needed to serve Fast Track and the operator portal, under documented operator instructions in a DPA.

  Connector A: Supabase Auth Identity Connector

Operator hosts their own Supabase project. NeoStrike validates the JWT on every API call and stores a minimum player profile for Fast Track and portal operations.

2.1 What Supabase Auth Provides
Supabase Auth is an open-source JWT-based authentication server (GoTrue) that operators can run on their own Supabase cloud project. It handles player registration, email verification, login, password reset, session refresh, and social OAuth. The operator's Supabase project is entirely isolated — it has its own database, its own JWT signing keys, and its own data residency. NeoStrike has no access to it.

2.2 JWT Validation Architecture
Every NeoStrike API call must include the player's Supabase Auth JWT in the Authorization: Bearer header. NeoStrike validates this JWT locally using the operator's Supabase JWKS endpoint — no round-trip to Supabase Auth on every request. The JWKS endpoint is cached in Upstash Redis (TTL: 1 hour) per tenant.

JWT Validation Steps

PERFORMANCE: JWT validation must complete in < 5ms p99 for steps 1–5. Step 6 (KYC gate) adds a Redis lookup — total including KYC check must remain < 10ms p99. JWKS caching is mandatory to achieve this.

2.3 Player Identity Model in NeoStrike
NeoStrike uses the Supabase Auth JWT sub claim (a UUID) as the canonical player_id throughout the system — it is the foreign key in all wallet records, bonus instances, payment transactions, and event logs. In addition, NeoStrike stores a minimum player profile in the player_profiles table: email address, country, registration timestamp, and display name. These fields are required to publish meaningful player events to Fast Track (which needs email and country for campaign segmentation) and to support operator portal player search. This profile is populated on first authenticated API call from the JWT claims and is personal data under GDPR — NeoStrike processes it as a data processor under the operator's DPA.

// JWT sub claim used as player_id throughout NeoStrike
{ "sub": "11111111-2222-3333-4444-555555555555",  // player_id in NeoStrike
  "iss": "https://xyzproject.supabase.co/auth/v1",
  "exp": 1714571490,
  "role": "authenticated" }

2.5 Registration Event
When a new player first authenticates with NeoStrike (first valid JWT received for a previously unseen sub), NeoStrike automatically creates a wallet record for that player and publishes a REGISTRATION event to RabbitMQ for Fast Track. No explicit registration endpoint is required — wallet creation is lazy and triggered by first authenticated API call.


  Connector B: Sumsub KYC/AML Connector

Operator contracts Sumsub directly. KYC verification happens in Sumsub's platform. NeoStrike receives only a KYC status signal via webhook — document scans and biometric data never enter NeoStrike.

3.1 What Sumsub Provides
Sumsub is the leading iGaming KYC/AML provider, named a Gartner Magic Quadrant Leader for Identity Verification in 2025 for the second consecutive year. It handles document verification (passport, driving licence, national ID), liveness detection, AML screening, PEP/sanctions checks, and ongoing monitoring. Critically for NeoStrike's architecture, Sumsub delivers verification results via webhooks — NeoStrike only ever sees the outcome, never the documents.

3.2 KYC Status Model
NeoStrike maintains a single kyc_status field per player in the player_wallets table. This field is updated exclusively by Sumsub webhooks. It is the only KYC data NeoStrike ever holds.

Status
Meaning
Wallet Operations Permitted
NOT_STARTED
Player has not begun KYC
Configurable per tenant. Typically: deposits allowed up to threshold, withdrawals blocked.
PENDING
KYC submitted, awaiting Sumsub decision
Same as NOT_STARTED. Configurable.
VERIFIED
Sumsub GREEN result — identity confirmed
All operations permitted. KYC gate lifted.
REJECTED_RETRY

REJECTED_FINAL

BLOCKED

All operations blocked. PLAYER_BLOCKED event to Fast Track.

3.3 Sumsub Webhook Integration
Sumsub calls NeoStrike's webhook endpoint when a verification decision is made. NeoStrike verifies the webhook signature, maps the Sumsub result to its kyc_status enum, updates the player record, and publishes a KYC_UPDATE event to RabbitMQ.

Sumsub Webhook Payload (inbound to NeoStrike)
POST /api/v1/kyc/webhook/sumsub
x-payload-digest: {HMAC-SHA256 signature}
{
  "type":          "applicantReviewed",
  "applicantId":   "sumsub_applicant_abc123",
  "externalUserId":"11111111-2222-3333-4444-555555555555",  // player sub UUID
  "reviewResult": {
    "reviewAnswer":  "GREEN",           // GREEN | RED
    "rejectType":    null,              // FINAL | RETRY (RED only)
    "reviewRejectDetails": []
  }
}

Webhook Processing Logic
// NeoStrike maps Sumsub result → kyc_status
GREEN              → VERIFIED
RED + RETRY        → REJECTED_RETRY
RED + FINAL        → REJECTED_FINAL  (also sets wallet_blocked=true)
applicantOnHold    → PENDING
// Post-verification adverse finding:
applicantReset + BLOCKED sanctions → BLOCKED (wallet_blocked=true)

CRITICAL — externalUserId: Operators must configure Sumsub to set externalUserId to the player's Supabase Auth UUID (the sub claim) when creating a Sumsub applicant. This is the only way NeoStrike can correlate a Sumsub webhook to a player wallet. This must be documented clearly in the F10 integration guide.

3.5 KYC Initiation Flow
NeoStrike provides a KYC initiation endpoint that the operator's frontend calls to start the Sumsub verification flow. NeoStrike creates a Sumsub access token for the player and returns it to the frontend. The frontend renders the Sumsub Web SDK directly — the operator is the data controller for everything that happens inside the SDK.

POST /api/v1/kyc/initiate
Authorization: Bearer {player_jwt}
{}  // No body required — player_id extracted from JWT sub

// NeoStrike calls Sumsub server-to-server:
POST https://api.sumsub.com/resources/accessTokens
  ?userId={player_sub_uuid}
  &levelName={tenant_kyc_level}
X-App-Token:  {SUMSUB_APP_TOKEN}
X-App-Access-Sig: {HMAC signature}

// NeoStrike response to frontend:
{ "sumsub_token": "eyJhbGciOiJS...",  // Short-lived Sumsub SDK token
  "kyc_status":   "NOT_STARTED" }

The operator's frontend passes sumsub_token to the Sumsub Web SDK. The player completes verification entirely within Sumsub's hosted UI. NeoStrike is not in the verification flow — it waits for the webhook.

3.6 Operator Setup — Sumsub Connector
What the Operator Does (One Time)
	•	Create a Sumsub account and configure a verification flow in Sumsub back-office for their jurisdiction (e.g. UKGC requires document + liveness; MGA requires document only at registration)
	•	In Sumsub back-office: create webhook pointing to https://api.neostrike.io/api/v1/kyc/webhook/sumsub, note the webhook secret
	•	In NeoStrike Tenant Portal > KYC > Sumsub: enter SUMSUB_APP_TOKEN, SUMSUB_SECRET_KEY, SUMSUB_WEBHOOK_SECRET, KYC_LEVEL_NAME

	•	Test with Sumsub's sandbox — Tenant Portal shows received webhook events in real time

5. Database Schema Changes

F10 adds schema changes to NeoStrike's existing Supabase tables. The principle: store only what is operationally necessary — the minimum player profile for Fast Track and portal, plus KYC status for wallet gating. Never store document scans, biometrics, or payment card data.

5.1 player_profiles (NEW TABLE — minimum player profile)
This table stores the minimum player profile NeoStrike needs to publish meaningful Fast Track events (which require email and country for campaign segmentation) and to support operator portal player search. It is populated on the player's first authenticated API call using claims from the Supabase Auth JWT. This data is personal data under GDPR. NeoStrike processes it as a data processor under the operator's DPA. The operator is the data controller and instructs NeoStrike on retention and deletion via the DPA.
player_profiles  (GDPR basis: legitimate interest / operator DPA)
  player_id        VARCHAR(255) PK    -- Supabase Auth sub UUID
  tenant_id        UUID FK
  email            VARCHAR(255)       -- from JWT email claim. PII. Encrypted at rest.
  display_name     VARCHAR(255)       -- from JWT name claim. PII. Encrypted at rest.
  country          CHAR(2)            -- ISO 3166-1. From JWT custom claim or operator config.
  language         CHAR(5)            -- IETF BCP 47 e.g. en-GB. For Fast Track segmentation.
  registered_at    TIMESTAMPTZ        -- First authenticated API call timestamp.
  last_seen_at     TIMESTAMPTZ        -- Updated on every valid JWT. Used for LOGIN event.
  deleted_at       TIMESTAMPTZ        -- Soft delete for GDPR right-to-erasure requests.

5.2 player_wallets — New Columns
ALTER TABLE player_wallets ADD COLUMNS:
  kyc_status       ENUM(NOT_STARTED,PENDING,VERIFIED,
                        REJECTED_RETRY,REJECTED_FINAL,BLOCKED)
                   DEFAULT 'NOT_STARTED'
  wallet_blocked   BOOLEAN DEFAULT FALSE    -- hard block flag
  sumsub_applicant_id  VARCHAR(255)         -- Sumsub reference (opaque ID, not PII)
  kyc_verified_at  TIMESTAMPTZ              -- timestamp of VERIFIED status

5.2 kyc_events (immutable audit log)
kyc_events
  id               UUID PK
  tenant_id        UUID FK
  player_id        VARCHAR(255)    -- Supabase Auth sub UUID only
  event_type       ENUM(STATUS_CHANGED,WEBHOOK_RECEIVED,
                        INITIATION_CREATED,GATE_BLOCKED)
  previous_status  VARCHAR(64)
  new_status       VARCHAR(64)
  sumsub_review_answer  VARCHAR(32)   -- GREEN / RED (from webhook)

PII NOTE: kyc_events stores no KYC document data or biometrics. sumsub_applicant_id is an opaque Sumsub-generated UUID. sumsub_review_answer and sumsub_reject_type are operational enums. The player's documents, selfie, address, and personal identity data remain exclusively in Sumsub's systems. The player_id field in this table is the Supabase Auth UUID — it references the player_profiles table where the minimum profile (email, country) is stored under the operator's DPA.

9. Acceptance Criteria

9.1 Supabase Auth Connector
	•	Valid RS256 JWT passes all 6 validation steps and request proceeds within 10ms p99
	•	Expired JWT returns 401 token_expired
	•	JWT with wrong issuer returns 401 invalid_issuer
	•	HS256 JWT returns 401 invalid_algorithm — consistent with F8 security policy
	•	First API call for an unknown player_id creates player_wallet record and publishes REGISTRATION event to RabbitMQ
	•	JWKS cached in Redis — second validation request does not call Supabase JWKS endpoint

9.2 Sumsub KYC Connector
	•	POST /kyc/initiate returns valid Sumsub SDK token within 500ms
	•	Sumsub GREEN webhook correctly sets kyc_status = VERIFIED and publishes KYC_UPDATE event within 50ms
	•	Sumsub RED + FINAL webhook sets kyc_status = REJECTED_FINAL, wallet_blocked = true, publishes PLAYER_BLOCKED event
	•	Sumsub RED + RETRY webhook sets kyc_status = REJECTED_RETRY — wallet not blocked
	•	Webhook with invalid HMAC signature returns 401 — kyc_status not updated
	•	Duplicate webhook (same applicantId + same reviewAnswer) processed idempotently — kyc_status not changed twice

9.3 KYC Gating
	•	Player with VERIFIED status: all gated operations proceed normally
	•	Player with REJECTED_FINAL: all wallet operations return 403 player_blocked regardless of gating configuration
	•	Withdrawal by NOT_STARTED player returns 403 kyc_required (default gate)
	•	UKGC-licensed tenant with default game_launch gate = VERIFIED: game session blocked until VERIFIED

9.4 Full Journey
	•	New player: registration → KYC → deposit → bonus → game play completes end-to-end without errors
	•	All 4 RabbitMQ events published in correct order: REGISTRATION, KYC_UPDATE, DEPOSIT, BONUS_AWARDED
